# Microtonal Isomorphic Keyboard — Design

**Date:** 2026-06-17
**Status:** Approved (pre-implementation)

## Summary

A zero-build single-page web app that renders a **generalized Wicki-Hayden
isomorphic hex keyboard** for any N-tone equal temperament (N-TET / N-EDO),
where `N` ranges from 2 to 24 (default 12). The keyboard spans **two octaves**
of pitch above a user-set base frequency. Each hex shows its frequency and step
index, is colored by pitch class, and plays its pitch when clicked via the Web
Audio API.

The "twist" is that the keyboard's shape, key count, and arrangement recompute
live as `N` changes — a faithful, theory-grounded visualization of how
equal-temperament keyboards generalize beyond 12.

## Background & rationale

The familiar piano (7 white + 5 black keys) is **specific to 12-TET**: the white
keys are the diatonic scale generated by stacking fifths, and that structure is a
*MOS scale* (Moment of Symmetry, Erv Wilson). It is also **not isomorphic** — a
C-major chord and a C#-major chord have different shapes.

Real microtonal instruments solve this with **isomorphic / generalized
keyboards** (Bosanquet 1875 → Erv Wilson → the modern Lumatone). On these, a
given interval or chord has the **same shape in every key and every tuning**
(transpositional + tuning invariance). This app implements the **Wicki-Hayden**
isomorphic layout, generalized to arbitrary N.

| Layout model | True 2D isomorphic (not a linear piano row) |
| Layout family | Generalized **Wicki-Hayden**, hexagonal keys |

## Core math

Let `N` = number of equal divisions of the octave, `base` = base frequency (Hz).

- **Best fifth:** `g = round(N × log₂(3/2))` steps  (`log₂(3/2) ≈ 0.585`)
- **Fourth:** `fourth = N − g`
- **Whole tone:** `tone = 2g − N`  (the horizontal Wicki step)

**Lattice (rhombic basis).** Each hex cell is addressed by integer coordinates
`(u, v)`:

- up-right neighbor = `+g` steps (a fifth)
- up-left neighbor  = `+fourth` steps (a fourth)

So a cell `(u, v)` has:

```
step(u, v)  = u·g + v·fourth
pitch(u, v) = base × 2^(step / N)
```

Note `g + fourth = N` (octave) and `g − fourth = 2g − N = tone`, so the
horizontal axis (right) is exactly a whole tone — consistent with Wicki-Hayden.

**Screen position** (up = higher pitch):

```
x ∝ (u − v)      // horizontal spread
y ∝ −(u + v)     // higher pitch drawn higher on screen
```

Hex polygons are packed at these positions (pointy-top or flat-top; see Rendering).

**Pitch window (grid extent).** The origin cell `(0, 0)` is the base frequency
(the origin/tonic is identified by `u===0 && v===0`, **not** by `step===0`, since
step 0 recurs at duplicates). Render cells whose `step` lies in `[0, 2N]` (two
octaves above base, inclusive of the two-octave note). Because the lattice is
isomorphic, the **same pitch appears at multiple `(u, v)` positions — these
duplicates are kept intentionally**; the repetition is the defining visual
feature of an isomorphic keyboard.

> **Note (refined during planning):** the band is *mathematically infinite*. The
> map `(u,v) → step` has a kernel along `(fourth, −g)`, so infinitely many
> `(u, v)` keep `step` in `[0, 2N]`. The renderer therefore enumerates a
> **deliberate finite display window** bounded by a fixed column count (the
> horizontal screen axis `u − v`); for each column the full in-window `v`-range
> is enumerated, so the window is *complete per column*. Widening the column
> budget only adds more columns of the (infinite) band — it is a display budget,
> not a completeness bound.

## Edge case: when the lattice cannot reach all N notes

> **Correction (2026-06-17, found during planning by adversarial review):** the
> original draft of this section claimed the fallback was needed *only* at N=2
> and N=4. That was wrong — it considered only the whole-tone collapse and
> missed the coprimality requirement below.

Two distinct conditions break the lattice:

1. **Whole-tone collapse:** `tone = 2g − N ≤ 0` makes the Wicki horizontal axis
   collapse (right neighbor duplicates the origin pitch). In range 2–24 this is
   only N=2 (`tone = 0`).

2. **Pitch-class coverage (the load-bearing one):** the lattice reaches a step
   `s = u·g + v·fourth`, and mod N that is `(u − v)·g (mod N)`. So the set of
   **reachable pitch classes is exactly the multiples of `gcd(g, N)`** — only
   `N / gcd(g, N)` of the N classes. When `gcd(g, N) > 1` the keyboard is
   permanently missing notes (e.g. 24-TET would show only the 12 even steps;
   15-TET only 5 of 15). A generator must be **coprime to N** to span the full
   scale, and the *best* fifth often is not.

**Verified fallback set** (union of both conditions across N=2…24):

```
N ∈ { 2, 4, 6, 10, 14, 15, 20, 21, 24 }
```

**Rule:** for any N in that set, fall back to a **single-row uniform layout** —
keys in a straight row where the right neighbor is `+1` EDO step, rendering steps
`0…2N`. This guarantees all N pitch classes are visible and playable. For all
other N the Wicki-Hayden hex lattice is used. This is the only conditional
branch in the layout logic. (Design alternative considered and rejected:
nudging the generator to the nearest coprime step would preserve hexes for all N
but detune the displayed "fifth" — we keep the true best fifth and accept a flat
row for these N.)

## Components / modules

Pure-logic modules are DOM-free and unit-testable in isolation.

| File | Responsibility | DOM? |
|------|----------------|------|
| `index.html` | Page shell, control bar markup, SVG container | — |
| `styles.css` | Layout, control + hex styling | — |
| `src/tuning.js` | `g`, `fourth`, `tone`, `step → pitch`, pitch-class | No |
| `src/layout.js` | Lattice enumeration, pitch-window clip, `(u,v) → (x,y)`, degeneracy fallback | No |
| `src/render.js` | Draw SVG hexes, labels, pitch-class coloring | Yes |
| `src/audio.js` | Web Audio: oscillator + envelope, waveform, master gain, voice management | Yes (AudioContext) |
| `src/main.js` | Wire controls → recompute → render; pointer events → audio | Yes |

**Data flow:** control change → `main.js` reads state `{base, N, waveform,
volume}` → `tuning`/`layout` produce a list of `{u, v, step, pitch, pitchClass}`
cells → `render` paints SVG → pointer events on hexes call `audio.play/stop`.

## Rendering & coloring

- Each cell is an SVG `<polygon>` hexagon positioned from `(x, y)`.
- **Color by pitch class** = `step mod N`: hue = `(pitchClass / N) × 360°` so all
  octave-equivalents share a hue. Origin/tonic hex gets a distinct stroke/marker.
- Labels per hex: **frequency** (Hz, 1 decimal) and **step index** (`step mod N`).
  Label font scales down gracefully as hex count grows at high N.

## Interaction & audio

- **Pointer down** on a hex starts a voice; **pointer up / leave** releases it.
  Polyphonic (multiple simultaneous voices).
- Synth per voice: one `OscillatorNode` (selected waveform) → `GainNode`
  (attack/release envelope) → master `GainNode` (volume) → destination.
- A single shared `AudioContext`, created/resumed on first user gesture (browser
  autoplay policy).

## UI controls (top bar)

- **Base frequency (Hz):** number input, default `440`, quick presets `220` / `440`.
- **Divisions (N):** slider + number, range `2–24`, default `12`, live.
- **Waveform:** select — sine / triangle / square / saw.
- **Volume:** slider (master gain).

All changes recompute and re-render immediately.

## Testing

Lightweight unit tests for the pure modules (`node --test` or a tiny runner; no
build dependency):

- `tuning.js`: `g/fourth/tone` for the worked-example table; `step → pitch`
  (e.g. step `N` = exactly `2×base`); pitch-class wrap.
- `layout.js`: pitch-window membership (only `0 ≤ step ≤ 2N`); origin at
  `(0,0)`; presence of intentional duplicate pitches; **degeneracy fallback at
  N=2 and N=4** produces the uniform single row.
